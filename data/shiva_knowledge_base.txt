Q10 — How Do You Approach Agentic AI System Design?
Topic tags: Agentic AI, System Design, LangGraph, MCP, RAG, GraphRAG, ReAct, Tool Use, Reliability
Core Design Philosophy
Agentic AI systems are fundamentally different from traditional ML pipelines — they involve non-deterministic, multi-step decision loops with external tool access. The design must account for reliability, observability, and graceful degradation from the start.
Step 1: Define the Agent Reasoning Backbone
Choose the right orchestration framework based on workflow complexity. LangGraph is preferred for complex, stateful, multi-step workflows (like CSM-Rovo) because it provides explicit state management, branching logic, and checkpointing.
LangChain is suitable for simpler, linear agentic workflows — used at Autodesk for incident resolution agents.
Define the reasoning pattern upfront: ReAct (Reasoning + Acting) for tool-heavy agents; CoT (Chain-of-Thought) for reasoning-heavy tasks.
Step 2: Design Tool Orchestration
Use Model Context Protocol (MCP) for dynamic tool registration and invocation — this decouples the agent logic from specific tool implementations, making the system extensible without re-engineering the agent.
Each tool should have a well-defined schema, idempotency where possible, and timeout/retry policies.
Step 3: Build the Knowledge Retrieval Layer
For factual grounding: implement RAG with a VectorDB for dense semantic retrieval.
For complex, interconnected knowledge: use GraphRAG — retrieval traverses a knowledge graph to surface related context that flat document retrieval would miss.
Hybrid retrieval (LlamaIndex semantic pipelines + structured queries) gives the best response relevance — demonstrated 35% improvement at Atlassian.
Step 4: Handle Agent Failure Modes
Implement LangGraph checkpointing so agents can resume from known-good states after tool failures or context overflows.
Add MCP-level fallback strategies: if a primary tool fails, the agent should gracefully degrade to a secondary source rather than halting.
Set hard limits on agent loop iterations to prevent runaway execution.
Step 5: Observability and Evaluation
Use LangSmith for tracing agent execution chains — see exactly which tool was called, with what input, and what was returned at every step.
Instrument key metrics: tool call success rate, latency per step, context window usage, hallucination rate.
Build offline evaluation datasets to regression-test agent behavior before each deployment.
Q11 — How Do You Approach LLM Evaluation and Quality in Production?
Topic tags: LLM Evaluation, Model Quality, Production AI, Hallucination, LangSmith, Monitoring, Metrics
The Problem with LLM Evaluation
Unlike classical ML (where accuracy on a held-out test set is sufficient), LLM quality in production is multi-dimensional: factual correctness, response relevance, tone, latency, safety, and consistency all matter. Each dimension requires a different evaluation strategy.
Offline Evaluation (Pre-Deployment)
Build golden datasets: curated input-output pairs representing the full range of expected queries including edge cases, adversarial inputs, and high-stakes scenarios.
Use LLM-as-judge patterns: a separate model (e.g., GPT-4) evaluates the target model's responses against the golden dataset, scoring on faithfulness, relevance, and completeness.
Implemented model evaluation and optimization workflows for production AI systems at Thomson Reuters before any model was promoted to production.
Online Evaluation (Post-Deployment)
Use LangSmith to trace agent and LLM execution in production — each turn is logged with full context, tool calls, retrieved chunks, and final response.
Monitor response latency percentiles (P50, P95, P99) per model and per workflow step.
Track tool call success rates for agentic workflows — a degrading tool will cascade into poor agent responses.
Human-in-the-loop feedback loops: at Atlassian, 10k+ AI-assisted conversations provided implicit signal on resolution quality.
Retrieval Quality (RAG-Specific)
Measure retrieval precision and recall: are the right chunks being retrieved for each query?
At Atlassian, hybrid semantic retrieval via LlamaIndex improved response relevance by 35% — measured by comparing output quality against a baseline single-retriever system.
Model Optimization Cycle
Identify failure modes from LangSmith traces and production logs.
Update golden datasets with new failure cases, re-run offline evaluation.
Tune prompts, retrieval strategies, or swap models based on evaluation delta.
Gate production promotion behind a minimum quality threshold on the full evaluation suite.

Q19 — What Is Your Approach to RAG System Design?
Topic tags: RAG, Retrieval Augmented Generation, VectorDB, LlamaIndex, GraphRAG, Chunking, Embeddings, Hybrid Retrieval
RAG Design Philosophy
Production RAG systems require careful design at every layer: document processing, chunking strategy, embedding model selection, retrieval architecture, re-ranking, and response generation. Each layer has significant quality impact.
Document Processing and Chunking
Chunk size and overlap must be tuned to the content type: small chunks for precise fact retrieval, larger chunks for contextual understanding. Overlap prevents context loss at chunk boundaries.
Metadata enrichment: attach source, date, section, and entity tags to each chunk for filtered retrieval.
Retrieval Architecture
